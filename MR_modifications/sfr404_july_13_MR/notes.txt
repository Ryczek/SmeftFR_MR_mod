I kept doing more basic changes:

1. I realized the idea of "dim8 fields" for dim8 operators, i.e. without normalization factors. It required much more changes that I initially though of - one needs to define second copy of gauge groups with effective couplings, than define also copy of ALL fields, also fermions, to have dim8 fields coupling only to dim8 groups etc. Also representations must be distinguished, new groups must have indices SU2D8, SU2W8, Colour8 etc, otherwise FeynRules does stupid things or hangs forever. Eventually all fields are expressed in terms of physical fields which remained the same, so all these complication are internal only and invisible to users.

Then I reviewed our dim8 operator set, and discovered some problems - e.g. in some 4 field tensor operators it was

ExpandIndices[.... ] aux

and should be

ExpandIndices[.... aux]

For SU2 it was harmless as ExpandIndices was done earlier. For gluons it was not done, and with this typo gluons remained in operators as unphysical fields, without transition. Anyway, it seems that this transition was done later when Feynrules calculated vertices from Lagrangian, so no real bug, but due to luck this time.

Then I wrote small Perl script (in main dir, name is dim8.sh) which on disk automatically replaced all "normal" quantities in definitions of dim8 operators by "8" fields and other "8" quantities. It seems to work, but I tried few "new" dim8 operators only. Pleasy try some, too and eventually compare the CPU time with previous versions. Should be significant in some cases, like WWWW quadruple tensor.

Even now some operators seem undoable: 4 gluon tensor G4n1 takes 20 min in smeft_fr_init and 1.5h to generate UFO - even if there are no flavor indices. I guess symmetrization of 8 gluon vertices takes forever. Later it runs in MG5, with the warning "non-QCD gluon interactions". But G4n9, with some exotic symmetric structure couplings and Gtilde tensot, simply after 1/2 hour hangs completely my PC, I need to cut the power to restart - probably fully exceeding PC memory?

Anyway, this is done, just need to be checked.

2. As I said, it would be better to load to memory only operators on the OpList. As an example, I created dir lagrangian/Dim8Class12 and put there all 3 operators from this class as files ephi5.fr  qdphi5.fr  quphi5.fr. Then in smeft_io.m one should replace line

Get[ FileNameJoin[{SMEFT$Path, "lagrangian", "212_psi2phi5.fr"}] ];

by

SMEFT$Dim8Class12 = {"lephi5", "quphi5", "qdphi5"};

Get[ FileNameJoin[{SMEFT$Path, "lagrangian", "Dim8Class12", # <> ".fr"}] ] & /@
  Intersection[SMEFT$Dim8Class12, SMEFT$OperatorList];

and only the useful operators are loaded. This is very simple as long as we split current files into subdirs with separate operators, should be a great job for AI :-)
